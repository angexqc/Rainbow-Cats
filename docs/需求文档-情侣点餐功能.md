## 1. 项目背景

Rainbow Cats 是一款基于微信云开发的情侣互动小程序，目前具备任务系统、商城系统、仓库系统和积分系统。为增强情侣间的互动性和便利性，现需开发「情侣点餐」功能，让情侣双方可以共享菜单、远程点餐，并在确认后实时推送通知对方。

---

## 2. 功能概述

本功能包含四个核心模块：
1. **情侣关联** - 配对/绑定情侣关系，确定"对方"身份
2. **菜单管理** - 双方共享菜单，可上传、编辑、删除菜品
3. **点餐选菜** - 从共享菜单中选择菜品，购物车管理
4. **订单确认推送** - 确认点餐后扣除积分并推送通知对方

## 3. 详细需求

### 3.1 情侣关联模块

#### 3.1.1 配对机制设计

**配对流程**：
```
用户A 生成配对码 → 发送给用户B → 用户B 输入配对码 → 确认配对 → 建立绑定关系
```

**配对方式选择**：

| 方式 | 描述 | 优先级 |
|------|------|--------|
| 方式A | 6位数字配对码（推荐） | P0 |
| 方式B | 微信分享链接（可选） | P1 |
| 方式C | 扫码绑定（可选） | P2 |

**配对码规则**：
- 长度：6位数字
- 有效期：24小时
- 使用后立即失效
- 可重新生成

#### 3.1.2 情侣绑定状态

| 状态 | 描述 | 功能可用性 |
|------|------|-----------|
| 未绑定 | 用户未绑定情侣 | 仅可查看/编辑自己的菜品，无法点餐 |
| 已绑定 | 双方已确认配对 | 所有功能可用 |
| 解绑中 | 一方申请解绑中 | 部分功能限制 |
| 已解绑 | 配对关系已解除 | 回到未绑定状态 |

#### 3.1.3 未绑定用户处理

**未绑定用户的功能限制**：
- ✅ 可以上传和编辑自己的菜品
- ✅ 可以下架/删除自己的菜品
- ❌ 无法查看对方上传的菜品
- ❌ 无法进行点餐操作
- ❌ 无法查看订单历史

**引导绑定**：
- 首次进入点餐页面时，显示绑定引导弹窗
- TabBar"点餐"入口处显示绑定状态标识

#### 3.1.4 配对页面

| 功能项 | 描述 |
|--------|------|
| 页面路径 | `/pages/Pair/` |
| 入口 | 首页"设置"或点餐页面绑定引导 |
| 生成配对码 | 点击生成6位数字码，显示二维码 |
| 输入配对码 | 输入对方提供的配对码 |
| 解绑功能 | 显示当前绑定信息，支持解绑 |
| 解绑确认 | 解绑需二次确认，双方都会收到通知 |

### 3.2 菜单管理模块

#### 3.2.1 菜单共享策略

**菜单共享规则**：

| 场景 | 菜品可见性 | 说明 |
|------|-----------|------|
| 已绑定情侣 | 双方共享 | 双方都能看到彼此上传的菜品 |
| 未绑定情侣 | 各自独立 | 只能看到自己上传的菜品 |
| 菜品下架 | 不可见 | 下架菜品在点餐页面不显示 |
| 菜品删除 | 历史保留 | 已删除菜品在历史订单中保留快照 |

#### 3.2.2 菜单权限控制

**菜品操作权限矩阵**：

| 操作 | 自己上传的菜品 | 对方上传的菜品 | 通用说明 |
|------|--------------|--------------|----------|
| 查看 | ✅ | ✅ | 已绑定情侣可见 |
| 编辑 | ✅ | ❌ | 只能编辑自己创建的 |
| 下架/上架 | ✅ | ✅ | 双方都可以操作 |
| 删除 | ✅ | ❌ | 只能删除自己创建的 |
| 点餐 | ✅ | ✅ | 双方都可以点对方上传的 |

**权限校验说明**：
- 所有操作在云函数中校验 OpenId
- 防止通过接口直接调用绕过前端限制

### 3.4 订单与积分模块

#### 3.4.1 积分扣除逻辑

**积分扣除时机**：订单确认下单时立即扣除

**积分扣除规则**：

| 场景 | 扣除方式 | 说明 |
|------|---------|------|
| A点餐给B | 从A的账户扣除 | 点餐人支付积分 |
| B点餐给A | 从B的账户扣除 | 点餐人支付积分 |
| 取消订单 | 退还积分 | 退还给点餐人 |
| 订单完成 | 不再扣除 | 已在创建时扣除 |

**余额检查**：
- 下单前检查点餐人积分余额是否充足
- 余额不足时显示提示，无法下单
- 提示文案："积分不足，当前余额XX，需YY"

#### 3.4.2 订单确认页面

| 功能项 | 描述 |
|--------|------|
| 页面路径 | `/pages/OrderConfirm/` |
| 展示内容 | 订单明细、总金额、备注信息、当前余额 |
| 余额校验 | 余额不足时无法确认下单 |
| 确认按钮 | 确认后扣除积分、创建订单、推送消息 |
| 二次确认 | 下单前弹出二次确认对话框 |
| 网络异常处理 | 扣积分失败时回滚订单 |

#### 3.4.3 订单状态流转

**订单状态定义**：

| 状态 | 英文标识 | 说明 | 可操作 |
|------|---------|------|--------|
| 待确认 | pending | 订单已创建，等待对方确认 | 点餐人可取消 |
| 已确认 | confirmed | 对方已确认订单 | 双方可取消 |
| 已完成 | completed | 订单完成（手动标记） | 只能查看 |
| 已取消 | cancelled | 订单已取消 | 只能查看 |

**状态流转图**：
```
创建订单 → [待确认 pending]
   ↓
对方确认 → [已确认 confirmed]
   ↓
完成点餐 → [已完成 completed]
   ↑
   ├─────────┘
   │
   │ 取消订单
   ↓
[已取消 cancelled]
```

**订单状态说明**：

1. **待确认 (pending)**
   - 触发条件：订单创建成功
   - 操作权限：点餐人可取消订单

2. **已确认 (confirmed)**
   - 触发条件：接收方点击"确认订单"
   - 操作权限：双方都可取消订单

3. **已完成 (completed)**
   - 触发条件：双方中任意一方点击"完成订单"按钮
   - 操作权限：无，只能查看
   - 建议：在实际用餐完成后手动标记

4. **已取消 (cancelled)**
   - 触发条件：订单被取消
   - 操作权限：无，只能查看
   - 积分处理：退还给点餐人

#### 3.4.4 订单取消与超时

**取消订单规则**：

| 订单状态 | 谁可以取消 | 积分处理 |
|----------|-----------|---------|
| 待确认 | 仅点餐人 | 全额退还 |
| 已确认 | 双方 | 全额退还 |
| 已完成 | 不可取消 | - |

**取消订单二次确认**：
- 点击取消按钮后弹出确认对话框
- 显示："确认取消订单？积分将退还给[点餐人]"

**订单超时处理**：

| 场景 | 超时时长 | 处理方式 | 优先级 |
|------|---------|---------|--------|
| 待确认超时 | 24小时 | 自动取消并退还积分 | P1 |
| 已确认超时 | 无 | 不自动取消 | - |

**超时实现**：
- 使用云函数定时触发器
- 每小时检查一次超时订单
- 超时后更新订单状态为 cancelled
- 同时调用 editCredit 退还积分

|------|---------|---------|

**站内消息补充**（可选）：

- 在订单历史页面顶部显示未读通知标识

- 通知内容包括：订单状态变更记录

#### 3.6.4 推送失败处理

| 失败原因 | 处理方式 | 重试策略 |
|----------|---------|---------|

| 用户取消订阅 | 不推送，记录日志 | 不重试 |
| 模板失效 | 记录日志，告警 | 不重试 |
| API限流 | 延迟重试 | 5分钟后重试1次 |
| 网络异常 | 记录日志 | 不重试 |

## 4. 数据模型设计

### 4.1 情侣关系集合 (CoupleRelation)

```javascript
{
  _id: String,              // 关系ID
  openIdA: String,          // 用户A的OpenId
  openIdB: String,          // 用户B的OpenId
  status: String,           // 状态（active、unbinding）
  pairCode: String,         // 当前配对码
  pairCodeExpire: Date,     // 配对码过期时间
  bindTime: Date,           // 绑定时间
  unbindTime: Date,         // 解绑时间（如果已解绑）
  updatedAt: Date           // 更新时间
}
```

**索引配置**:
```javascript
{ openIdA: 1, status: 1 }
{ openIdB: 1, status: 1 }
{ pairCode: 1 }
```

### 4.2 菜单集合 (MenuList)

```javascript
{
  _id: String,              // 菜品ID
  _openid: String,          // 创建者OpenId
  title: String,            // 菜品名称
  image: String,            // 菜品图片URL
  desc: String,             // 菜品描述
  credit: Number,           // 价格/积分
  category: String,         // 分类（main、drink、dessert、other）
  available: Boolean,       // 是否上架
  star: Boolean,            // 是否星标
  date: Date,               // 创建时间
  updatedAt: Date          // 更新时间
}
```

**索引配置**:
```javascript
{ _openid: 1, date: -1 }
{ category: 1, available: 1, date: -1 }
```

### 4.3 订单集合 (OrderList)

```javascript
{
  _id: String,              // 订单ID
  creatorOpenId: String,    // 创建者OpenId（点餐人）
  partnerOpenId: String,    // 对方OpenId（接收人）
  items: Array,             // 菜品列表（快照）
  items: [{
    menuId: String,         // 原菜品ID
    title: String,          // 菜品名称（快照）
    image: String,          // 菜品图片（快照）
    desc: String,           // 菜品描述（快照）
    credit: Number,         // 单价（快照）
    count: Number           // 数量
  }],
  totalCredit: Number,      // 总金额
  remark: String,           // 备注信息
  status: String,           // 状态（pending、confirmed、completed、cancelled）
  date: Date,               // 下单时间
  confirmTime: Date,        // 确认时间
  completeTime: Date,       // 完成时间
  cancelTime: Date,         // 取消时间
  cancelledBy: String       // 取消者OpenId
}
```

**索引配置**:
```javascript
{ creatorOpenId: 1, date: -1 }
{ partnerOpenId: 1, date: -1 }
{ status: 1, date: -1 }
{ date: -1 }  // 用于超时扫描
```

## 5. 交互流程

### 5.1 情侣配对流程

```
用户A → 进入配对页面 → 生成6位配对码 → 发送给用户B
                                               ↓
用户B ← 输入配对码 ← 确认配对 ← 点击"添加绑定"按钮
   ↓
验证配对码 → 建立绑定关系 → 双方绑定成功 → 显示对方信息
```

### 5.2 解绑流程

```
用户 → 进入配对页面 → 点击"解除绑定" → 二次确认
                                                    ↓
                                                更新关系状态
                                                    ↓
通知对方 → 推送解绑消息 → 双方关系解除 → 功能受限
```

### 5.3 上传菜单流程

```
用户 → 点击TabBar"点餐" → 点击悬浮添加按钮 → 填写菜品信息
                                                          ↓
                                                  上传图片（自动压缩）
                                                          ↓
选择分类 → 填写价格 → 选择预设（可选） → 确认提交
                                                          ↓
                                        云函数验证 → 保存到数据库 → 跳转回列表页
```

### 5.4 点餐流程

```
用户 → 点击TabBar"点餐" → 浏览菜品（双方共享） → 调整数量
                                                          ↓
                                                  点击购物车弹窗
                                                          ↓
                                          填写备注（可选） → 检查余额
                                                          ↓
                                                  确认下单（二次确认）
                                                          ↓
                                        扣除积分 → 创建订单 → 推送消息给对方
                                                          ↓
                                                  跳转到订单历史
```

### 5.5 订单确认流程

```
接收方 → 收到推送消息 → 点击进入小程序 → 查看订单详情
                                                          ↓
                                                  点击"确认订单"按钮
                                                          ↓
                                        更新订单状态 → 推送确认消息给点餐人
```

### 5.6 订单取消流程

```
用户 → 进入订单详情 → 点击"取消订单" → 二次确认
                                                     ↓
                                             退还积分给点餐人
                                                     ↓
                                             更新订单状态为"已取消"
                                                     ↓
                                             推送取消消息给对方
```

## 9. 边界情况处理

### 9.1 数据一致性

| 场景 | 处理方式 |
|------|---------|
| 扣积分成功但订单创建失败 | 调用 editCredit 回滚积分，记录错误日志 |
| 订单创建成功但推送失败 | 订单仍有效，记录推送失败日志 |
| 网络中断导致操作中断 | 重新加载页面时同步最新状态 |

### 9.2 菜品状态变化

| 场景 | 处理方式 |
|------|---------|
| 菜品被删除 | 历史订单保留快照，点餐页面不显示 |
| 菜品被下架 | 点餐页面不显示，购物车中显示灰色 |
| 菜品价格变更 | 点餐使用实时价格，历史订单不受影响 |
| 菜品图片变更 | 历史订单使用快照图片 |

### 9.3 订单异常处理

| 场景 | 处理方式 |
|------|---------|
| 重复提交订单 | 前端防抖+后端幂等性校验 |
| 订单不存在 | 显示友好提示，跳转订单列表 |
| 订单状态冲突 | 重新加载最新状态 |

### 9.4 并发冲突

| 场景 | 处理方式 |
|------|---------|
| 双方同时取消订单 | 第一次操作成功，第二次提示"订单已取消" |
| 同时确认和取消订单 | 数据库使用乐观锁，第一个操作成功 |
| 同时下单导致余额不足 | 第二次订单检查余额失败，提示"积分不足" |
